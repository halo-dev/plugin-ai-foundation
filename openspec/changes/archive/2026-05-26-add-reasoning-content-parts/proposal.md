## Why

Reasoning-capable models can emit hidden or visible thinking content that must be preserved as first-class generation data, especially when later tool calls require the assistant reasoning content to be sent back to the provider. The current text/tool part model cannot represent this content, which causes provider errors such as DeepSeek's `reasoning_content` round-trip requirement and leaves callers without a model-independent way to inspect or render reasoning.

## What Changes

- Add model-independent reasoning content parts for generated assistant output, message history, generation steps, and streaming deltas.
- Extend `GenerateTextResult`, `GenerationStep`, and `TextStreamPart` so callers can read `reasoning`, `reasoningText`, and reasoning token usage without provider-native types.
- Preserve provider reasoning metadata needed for follow-up requests, including provider-specific fields such as DeepSeek/OpenAI-compatible `reasoning_content`, while keeping the public API Halo-owned.
- Convert assistant reasoning parts back into provider messages during multi-step tool loops and caller-provided message history.
- Update the model test workbench to tolerate and optionally display reasoning stream parts separately from assistant answer text.
- Update documentation and tests for reasoning output, reasoning round-trip, and streaming protocol behavior.

### Non-Goals

- Do not add backward compatibility shims for prior internal stream part names or legacy chat APIs.
- Do not expose Spring AI or provider-native message classes in the public API.
- Do not implement full structured output, source parts, file parts, or response body persistence in this change.
- Do not attempt to infer or fabricate reasoning content when the provider does not return it.

## Capabilities

### New Capabilities

- `reasoning-content-parts`: Model-independent representation, streaming, round-trip, and UI handling for reasoning generated by language models.

### Modified Capabilities

- `ai-model-service`: Language model requests and results gain reasoning content, reasoning text aggregation, reasoning metadata, and provider round-trip behavior.
- `test-chat-streaming`: The test-chat SSE protocol gains reasoning stream parts and preserves answer text separately from reasoning text.
- `model-test-workbench`: The test workbench handles reasoning stream parts without mixing them into assistant answer text.

## Impact

- Public API module: new reasoning part types and additions to generation result, step, stream part, message part, and usage DTOs.
- App implementation: Spring AI/OpenAI-compatible response mapping, DeepSeek reasoning metadata preservation, multi-step tool loop message reconstruction, and request validation.
- Console endpoint and UI: generated OpenAPI client updates plus optional reasoning display in the model test workbench.
- Tests/docs: backend unit tests, frontend stream parser tests, and `dev/dev.md` examples for reasoning-capable models.
